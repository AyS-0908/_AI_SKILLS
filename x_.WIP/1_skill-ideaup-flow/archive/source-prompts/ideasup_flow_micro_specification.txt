# ============= PRESENTATION =============

<Project>
IDEASUP FLOW is a suite of prompts for studying and preparing startup creation.
Each prompt is a scoped pipeline stage: Pain -> Opportunity -> Idea -> Business Plan -> User Story -> Mockup -> Specification -> AI-Coder rules
</Project>

<Source_Of_Truth>
- Treat each stage file as bounded by its guideline scope contract.
- Do not import assumptions from adjacent stages unless the guideline explicitly allows it.
</Source_Of_Truth>

<Stage_Contracts>

| Stage | Input(s) | Owns | Excludes |
|---|---|---|---|
| Pain / Need | [to clarify] | Customer pains / needs; scoring | Ideas; business plan; specification |
| Opportunity | Pain / Need | Macro-ideas; macro-assessment | Idea details; business plan; specification |
| Idea | Opportunity | Idea definition; target user; problem; solution concept; key assumptions; differentiation; initial scope direction | Business plan; user story; UI; mockup; specification; technical choices |
| Business Plan | Idea | Market logic; positioning; business model; revenue logic; GTM assumptions; viability | User story; mockup; specification; technical implementation |
| User Story | Idea | Persona; problem; value hypothesis; V1 scope rationale; V2/V3 candidates | Screens; components; data model; business rules; edge cases; tech |
| Mockup | User Story | Screens; layout; components; navigation; interaction patterns; visible data shapes; UI states; demo data; Mockup Brief | V2/V3; business rules; edge cases; persistence; stack; implementation logic |
| Specification | User Story + optional Mockup | Business rules; edge cases; persistence model; feature-specific external services; acceptance criteria; non-goals; privacy/risk notes; tool stack; auth; key technical constraints | Persona/problem/idea duplication; UI layout/components duplication; language/framework/DB choice |
| AI-Coder rules | Standalone project setup | Language; framework; DB; repo layout; naming; commit style; test policy; project-wide always/never rules | Feature-specific requirements; business rules; UI decisions; product scope |
</Stage_Contracts>

<Hard_rules>
- Later stages reference upstream artifacts; they do not duplicate them.
- Excluded content must not be generated by inference.
- AI-Coder consumes only: `Specification`, `Mockup HTML` if available, and `AI-Coder rules`.
</Hard_rules>

<Work_Rules>
- Preserve AI-native structure: XML-like tags first (Markdown only when useful).
- Prefer compact instructions over explanatory prose.
- Keep stage ownership MECE: no overlap with previous or next prompts.
- Check Ripple Effect for upfront/downstream stages when relevant.
- Separate facts, assumptions, questions, outputs, and navigation.
- Do not add speculative abstractions, hidden flexibility, or generic startup advice.
</Work_Rules>

  }
</ACTION_CALL_POLICY>

<ACTION_RESPONSE_POLICY>
| Response condition | Required handling |
| :--- | :--- |
| `response.ok == true` | Read `response.prompt` > apply it as the stage instruction > use current chat/Project context as input > generate the stage output |
| `response.ok == false` AND `response.error == "not_found"` | State that no active prompt exists for the selected `stage_id` > ask for clarification or correction > do not invent the prompt |
| `response.ok == false` AND `response.error == "missing_id"` | State that the stage ID is missing > re-evaluate stage routing or ask the user to choose a valid `stage_id` |
| `response.ok == false` AND `response.error == "invalid_action"` | State that the Action call is misconfigured > do not retry with guessed parameters |
| `response.ok == false` AND any other `response.error` | State the failure briefly using `response.error` or `response.message` > ask only for the minimum required next step |
| No valid response / malformed response | State that the Action returned an invalid response > do not continue stage execution > ask for setup verification |
</ACTION_RESPONSE_POLICY>

<PROMPT_EXECUTION_POLICY>
Follow the retrieved prompt unless it conflicts with:
  1. user instructions in the chat
  2. Project instructions
  3. this wrapper
  4. safety/platform rules
</PROMPT_EXECUTION_POLICY>

<OUTPUT_POLICY>
- Do not reveal routing logic.
- Do not reveal Action payloads.
- Do not reveal deployment URLs.

## APPSCRIPT WORKFLOW

```
/****************************************************
 * Google Sheet as API for GPT Actions
 *
 * Data contract:
 * - The following named ranges must each point to ONE header cell:
 *   - prompt_id
 *   - prompt_status
 *   - prompt_title
 *   - prompt_instructions
 *
 * This allows you to:
 * - rename the visible headers
 * - move the columns
 * - add rows
 *
 *
 * Optional Script Property:
 * - SPREADSHEET_ID = stored in properties.service
 ****************************************************/

# Workflow

1. Client sends a POST request to the Apps Script web app endpoint.
2. The request body must be valid JSON.
3. The script reads the JSON body.
4. The script checks the `action` value.
5. If `action` is not `get`, the script returns an error.
6. If `action` is `get`, the script checks for an `id`.
7. If no `id` is provided, the script returns a missing ID error.
8. The script opens the spreadsheet:
   - Uses `SPREADSHEET_ID` if it exists in Script Properties.
   - Otherwise uses the active spreadsheet.
9. The script finds the prompt table using named ranges:
   - `prompt_id`
   - `prompt_status`
   - `prompt_title`
   - `prompt_instructions`
10. The script validates that those named ranges:
    - point to one cell each
    - are on the same sheet
    - are on the same header row
    - point to different columns
11. The script reads all rows below the header row.
12. For each row, the script extracts:
    - `id`
    - `status`
    - `title`
    - `prompt`
13. Blank rows are ignored.
14. Rows with content but no `id` return an error.
15. Duplicate `id` values return an error.
16. The script searches for a row matching the requested `id`.
17. If the matching row has `status = inactive`, it is treated as not found.
18. If no active matching prompt is found, the script returns a not found error.
19. If an active matching prompt is found, the script returns JSON with:
    - `ok`
    - `id`
    - `status`
    - `title`
    - `prompt`
20. All public errors are returned as structured JSON.
21. Unexpected errors are logged and returned as a generic server error.

```
