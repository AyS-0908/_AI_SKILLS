# Specification of `audit-it` Claude Skill

## 1. OBJECTIVE:

CREATE A CLAUDE.AI *SKILL* TO AUDIT TECHNICAL DOCUMENTATION (PRD, SPECIFICATION, CODE): the audit is a full framework with explicit reference frame, completeness checks, and testability checks.

**SKILL CONSTRAINTS (MUST HAVE):**
  - Minimize **tokens consumption**
  - Maximize **audit reliability**
  - Support **large specifications (~120k characters or more)**

**HIGH-LEVEL ARCHITECTURE:**
3-Phase / 4-Turn Pipeline

```
INPUT SPEC
     │
     ▼
Phase 1 — EXTRACT
(1–2 LLM turns)
     │
     ▼
Script — COMPILE
(no LLM)
     │
     ▼
Phase 2 — ANALYZE
(1 LLM turn)
     │
     ▼
Phase 3 — REPORT
(1 LLM turn)
     │
     ▼
FINAL AUDIT REPORT
(machine-readable fixes for downstream AI-coder)
```

---
## 2. PHASE 1- EXTRACT:

**PURPOSE:** 
Extract **structured artifacts** from the specification while the **full specification is still in context**.

### 0. AUDIT FRAME:

**OBJECTIVE:** 
Audit reference frame from the provided material; this audit frame becomes the reference baseline for later findings.
NB: `AUDIT_FRAME` extraction is part of **Phase 1 — EXTRACT** (it MUST NOT create an additional standalone phase or extra LLM turn).

**`AUDIT_FRAME` CONTENT**  
- audited artifact type
- product objective
- target user or audience
- explicit scope
- explicit non-goals / exclusions
- technical stack or execution environment
- key constraints
- success / acceptance criteria when present
- `missing_frame_items`: any required frame element not found in the source.

If one of these elements is missing: MUST be recorded in `missing_frame_items` and may produce a specification-quality finding in Phase 2.

### 1. PRIORITY TAGGING RULE

**OBJECTIVE:**
Priority tagging in Phase 1 MUST be mechanical only.

**RULE 1: A section MUST be marked as priority if it contains any of the following:**
- state management or persistence logic
- concurrency or async operations
- cross-component dependencies
- user input handling or validation
- error handling or recovery flows
- timing-sensitive operations.

**RULE 2: A section MUST also be marked as priority if it contains any of these strings, case-insensitive:**
- async
- await
- persist
- save
- db
- state
- mutex
- lock
- validate
- error
- retry
- timeout
- callback.

**RULE 3: The priority reason must be written into `priority_reason`.**

### 1. `segments.json`

**OBJECTIVE:**
Mechanical section map of the specification.

**IMPORTANT CONSTRAINTS:**
- segmentation must preserve the original section grouping whenever the specification already defines one
- no summarization, reinterpretation, renaming, or renumbering
- if an expected category is absent, it must be explicitly marked as `NOT_PRESENT`
- priority tagging must be mechanical and must include a reason

**EXAMPLE STRUCTURE:**
```json
{
  "segment_categories": {
    "Foundational Rules / Laws": ["§1.0", "§1.1"],
    "Single Sources of Truth (SSOTs)": ["§2.0"],
    "Interfaces & Contracts": ["§3.0"],
    "Runtime Execution & UI Flows": ["§4.0"],
    "Persistence & Storage and DB": ["§5.0"],
    "Output / Execution Artifacts": [],
    "NOT_PRESENT": ["API"]
  },
  "segments": [
    {
      "section_id": "§4.7",
      "title": "Execution Model",
      "line_start": 340,
      "line_end": 412,
      "category": "Runtime Execution & UI Flows",
      "priority": true,
      "priority_reason": "contains state management and persistence logic"
    }
  ]
}
```

**FIELDS:**

| Field | Meaning |
|---|---|
| segment_categories | mapping of high-level categories to section IDs |
| segment_categories.NOT_PRESENT | categories expected by the audit process but absent from the specification |
| section_id | original spec section identifier |
| title | original section title |
| line_start | starting line |
| line_end | ending line |
| category | original or mapped audit category |
| priority | whether section is high-risk |
| priority_reason | mechanical reason for priority tagging |

### 2. `contracts.json`

**OBJECTIVE:** 
Extract callable entities, their declared contracts, their consumption sites, and preserve a full explicit cross-reference map for the provided specification sections.

**NOTE:** 
The script compares signature fields; the LLM cites exact_signature_quote fields.

**EXAMPLE:**
```json
{
  "contracts": [
    {
      "name": "create_project",
      "name_quote": "create_project",
      "signature": "create_project(name: str) -> UUID",
      "exact_signature_quote": "create_project(name: str) -> UUID",
      "declared_return": "UUID",
      "declared_return_quote": "-> UUID",
      "declared_in": "§3.1",
      "consumption_sites": [
        {
          "location": "§4.7",
          "consumer_quote": "project_id = create_project(name)",
          "expected_type_quote": "project_id"
        }
      ]
    }
  ],
  "section_references": [
    {
      "source_location": "§4.7",
      "reference_text_quote": "See Section 3.1",
      "target_section": "§3.1",
      "exists_in_provided_chunks": true
    }
  ]
}
```

**FIELDS:**

| Field | Meaning |
|---|---|
| contracts | list of callable contract entries |
| contracts[].name | callable identifier |
| contracts[].name_quote | verbatim callable name |
| contracts[].signature | normalized storage form of the callable signature |
| contracts[].exact_signature_quote | verbatim signature quote from the specification |
| contracts[].declared_return | extracted return type label |
| contracts[].declared_return_quote | verbatim return-type quote from the specification |
| contracts[].declared_in | declaration section |
| contracts[].consumption_sites | list of consumption sites with verbatim quotes |
| contracts[].consumption_sites[].location | consumer section |
| contracts[].consumption_sites[].consumer_quote | verbatim call-site / usage quote |
| contracts[].consumption_sites[].expected_type_quote | verbatim expected / assumed type quote |
| section_references | full global cross-reference map for the provided sections |
| section_references[].source_location | source section |
| section_references[].reference_text_quote | verbatim cross-reference text |
| section_references[].target_section | referenced target |
| section_references[].exists_in_provided_chunks | mechanical existence check |

### 3. `tokens.json`

**OBJECTIVE:**
Extract raw token groups for lexical consistency analysis.

**IMPORTANT CONSTRAINTS:**
- No canonical token may be chosen in Phase 1
- Group tokens ONLY when:
    - they are explicitly defined as aliases in the specification, OR
    - they share identical identifier names case-insensitively across sections.
- Do NOT infer semantic equivalence
- If grouping confidence is uncertain, mark the group with `uncertainty_flag = true`.

**EXAMPLE:**

```json
{
  "concept_groups": [
    {
      "provisional_label": "project_id_family",
      "grouping_basis": "identical_identifier_case_insensitive",
      "uncertainty_flag": false,
      "variants": [
        {
          "value": "project_id",
          "variant_quote": "project_id",
          "location": "§2.4"
        },
        {
          "value": "projectId",
          "variant_quote": "projectId",
          "location": "§3.1"
        }
      ]
    }
  ]
}
```

**FIELDS:**

| Field | Meaning |
|---|---|
| provisional_label | temporary group label for storage only |
| grouping_basis | explicit_alias / identical_identifier_case_insensitive |
| uncertainty_flag | whether grouping is uncertain |
| variants | list of token occurrences |
| variants[].value | extracted token value |
| variants[].variant_quote | verbatim token quote |
| variants[].location | source section |

### 4. VERBATIM TRACEABILITY REQUIREMENT

**All Phase 1 artifacts MUST preserve enough verbatim evidence to support later contradiction analysis without globally reloading the specification.**

**MINIMUM REQUIREMENT:**
- every callable entry must include verbatim signature evidence
- every declared return must include a verbatim return quote
- every consumption site must include a verbatim usage quote
- every cross-reference must include a verbatim reference quote
- every token variant must include a verbatim token quote
- every uncertain grouping must be explicitly marked

**IF AN EXTRACTED ITEM CANNOT BE TIED TO A VERBATIM QUOTE:**
It MUST NOT be used as contradiction evidence in Phase 2.

### 6. EXTRACTION INTEGRITY CHECK (MANDATORY)

Before concluding Phase 1, the auditor MUST validate extraction completeness.

#### Step 1 — Segment Coverage Integrity

**PHASE 1 MAY PROCEED ONLY WHEN SEGMENT COVERAGE IS COMPLETE.**

The union of all extracted segments in `segments.json` MUST cover 100% of the specification sections or line ranges provided for audit.
If any section or line range is unassigned:
  - STOP
  - regenerate `segments.json`
  - do not continue to contracts or token extraction

#### Step 2 — Artifact Coverage Check

**VERIFY THAT THE FOLLOWING ARTIFACTS WERE SUCCESSFULLY GENERATED:**
- segments.json
- contracts.json
- tokens.json

**IF ANY ARTIFACT IS MISSING:**
STOP and regenerate Phase 1.

#### Step 3 — Suspicion Threshold

**IF ANY OF THE FOLLOWING CONDITIONS OCCUR, THE EXTRACTION MUST BE RERUN:**
- segments < 5
- callable signatures = 0 AND segments.json contains sections categorized as "Interfaces & Contracts"
- token groups = 0 AND segments count > 10
- cross references = 0 while multiple sections reference other sections

**REASON:**
Extraction failure or truncation likely occurred.

## 3. SCRIPT- `compile_artifacts.py`:

**PURPOSE:** 
Perform **mechanical validation** on extraction artifacts.

**IMPORTANT DESIGN CHOICE:**
  - The compile script supports only mechanical inconsistency detection and prioritization.
     - It does NOT assess product-goal alignment, completeness, testability, maintainability, or design intent.
     - Those dimensions remain LLM judgment tasks in Phase 2.
  - The script **never parses the original spec**.
       - The script MAY detect deterministic artifact-structure inconsistencies, but it MUST NOT decide:
          - analytical impact
          - contradiction validity
          - execution-model validity
          - report severity
       - It only parses **structured JSON artifacts generated by the LLM**, ensuring:
          - consistent format
          - reliable parsing
          - deterministic checks

---
### 1. INPUTS

```
/home/claude/audit_artifacts/

segments.json
contracts.json
tokens.json
```

---
### 2. OUTPUT

```
compiled_audit.json
```

**OUTPUT SCHEMA EXAMPLE:**

```json
{
  "conflict_candidates": [],
  "artifact_structure_candidates": [],
  "risk_hotspots": [],
  "artifact_stats": {
    "segments": 24,
    "contracts": 30,
    "concept_groups": 15
  },
  "parsing_warnings": []
}
```

**TYPICAL SIZE:** 
```1k – 2k tokens```

---
### 3. SCRIPT OPERATIONS
#### A. LOGIC

All checks MUST be **deterministic and ambiguity-free**.

**STEP 1. DUPLICATE FUNCTIONS**
  - Objective: Detect same function name with different signatures.
  - Script:
    ```
    group by function_name
    flag if signatures differ
    ```

**STEP 2. DANGLING REFERENCES**
  - Objective: Verify references target existing sections.
  - Script:
  ```
  For each entry in `contracts.json.section_references`, flag a dangling reference when:
   `target_section` is not equal to any `segments.json.segments[].section_id`
   and is not equal to any named artifact label explicitly recorded in Phase 1 artifacts.
  ```

**STEP 3. RETURN TYPE MISMATCH**
  - Objective: Simple string comparison.
  - Script: Compare
  ```
  declared return type
  vs
  consumer expected type
  ```

**STEP 4. TOKEN COLLISIONS**
  - Objective: Identify high collision risk.
  - Script: Concept groups containing **3+ variants**.

**STEP 5. ORPHAN FUNCTIONS**
  - Objective: Identify orphan functions
  - Script: Functions ```declared but never consumed```

**STEP 6. ARTIFACT STRUCTURE CHECKS**
  - Objective: Detect deterministic cross-artifact integrity issues.
  - Script:
     1. Flag a structure inconsistency when a callable in `contracts.json` cannot be linked to at least one valid segment location.
     2. Flag a structure inconsistency when `contracts.json.section_references[].target_section` resolves to neither:
        - a valid `segments.json.segments[].section_id`, nor
        - a named artifact label explicitly recorded in Phase 1 artifacts.
     3. Flag a structure inconsistency when a token variant location does not resolve to a valid segment location.

  - Output rule: Write each detected item into `compiled_audit.json.artifact_structure_candidates[]`
 with:
    - issue_type = "ARTIFACT_STRUCTURE_ERROR"
    - check_type
    - referenced_entity
    - source_artifact
    - evidence_fields


**STEP 7. RISK DENSITY**
  - Objective: Identify **audit hotspots**.
  - Script: Score per section:
  ```
  risk_score =
  (incoming references + outgoing references)
  × priority_weight
  ```

---
#### B. NON-FILTERING GUARANTEE

**OBJECTIVE:**
The script MUST NOT restrict the scope of the LLM analysis.

**RULE 1:**
Compiled_audit.json is used ONLY for:
- prioritizing investigation
- highlighting mechanical inconsistencies
- surfacing deterministic artifact-structure issues
- ranking high-risk sections


**RULE 2:** 
The LLM analyzer MUST still be allowed to inspect the full artifacts:
- segments.json
- contracts.json
- tokens.json

**CONSEQUENCE:**
compiled_audit.json = PRIORITY INDEX  
NOT = EXCLUSIVE INPUT

The analyzer must treat candidates in compiled_audit.json as a starting point, not a complete list of possible issues.

---
### 4. FAULT TOLERANCE

a) Each script operation MUST be independent.

b) If one operation fails because of a missing field, malformed JSON fragment, or unreadable artifact subsection:
- record a warning in `parsing_warnings[]`
- continue running the remaining mechanical checks
- still write `compiled_audit.json` if at least one artifact was parsed successfully

c) The script MUST NOT abort the full compile step solely because one mechanical check failed.

d) Phase 2 may use `parsing_warnings[]` as extraction-reliability context, but warnings MUST NOT be treated as contradictions by default.

**PURPOSE:** 
Perform **judgment analysis** on candidates detected by the script 

---
### 1. TURNS:

```
1 LLM turn
```

---
### 2. INPUTS:
**IMPORTANT:** The **full specification is NOT reloaded**.

The specification may only be accessed for:
- verbatim quote extraction when a required quote is absent from an artifact
- runtime sections required by execution-model validation
- source verification for artifact rows explicitly marked as incomplete, uncertain, or source-missing

Contradiction analysis MUST rely first on verbatim quote fields already stored in the artifacts.
- If a contradiction cannot be supported by artifact evidence or verified source text, it MUST NOT be reported.

**INPUTS:**

| Source              | Content                |
| ------------------- | ---------------------- |
| compiled_audit.json | primary analysis input |
| segments.json       | section metadata       |
| contracts.json      | callable details       |
| tokens.json         | terminology groups     |

---
### 3. ARTIFACT STRUCTURE IMPACT REVIEW:

**BEFORE STARTING MAIN ANALYSIS:**
Phase 2 MUST review `compiled_audit.json.artifact_structure_candidates[]` as mechanical extraction-reliability signals produced by the script.

**PHASE 2 RESPONSIBILITY:**
For each artifact-structure candidate, the auditor MUST determine whether it:
- is a minor extraction warning with limited analytical effect
- weakens contradiction analysis
- weakens execution-model validation
- blocks reliable interpretation for a specific claim or section

**REPORTING RULES:**
- record such items as issue type `ARTIFACT_STRUCTURE_ERROR`
- classify them under `EXTRACTION_WARNINGS`
- DO NOT treat them as contradictions by default
- DO NOT let the script decide severity or analytical consequence

**DEFAULT SEVERITY RULE FOR PHASE 3:**
- HIGH if the structure error invalidates contradiction or execution analysis for the affected claim or section
- otherwise MEDIUM

**OUTPUT CONSEQUENCE:**
Create a dedicated `EXTRACTION_WARNINGS` report section when one or more artifact-structure candidates materially affect audit reliability.

###  4. ARTIFACT-FIRST ANALYSIS RULE:
Phase 2 MUST perform reasoning using artifact files as the primary evidence layer.

**PRIMARY REASONING INPUTS:**
- compiled_audit.json
- compiled_audit.json.conflict_candidates[]
- compiled_audit.json.artifact_structure_candidates[]
- segments.json
- contracts.json
- tokens.json

### 5. PRIORITY-SECTION RULE

Sections marked `priority = true` in `segments.json` MUST be analyzed first.

For those sections:
- prefer depth over breadth
- do NOT suppress supported findings because of token discipline or informal severity caps

### 6. SECTION A — CONTRADICTION ANALYSIS:

**PRIMARY STRATEGY:**
Contradiction analysis MUST be candidate-first, using `compiled_audit.json.conflict_candidates[]` as the default investigation set.

**SECONDARY SWEEP (BOUNDED):**
A secondary artifact sweep is allowed only for:
- sections listed in `compiled_audit.json.risk_hotspots[]`
- sections where `priority = true`
- artifacts explicitly marked incomplete, uncertain, or source-missing

The auditor MUST NOT perform an unbounded full-artifact contradiction sweep.

**REQUIRED COVERAGE:**
- duplicate callable names with different signatures
- same callable used with incompatible expectations
- broken or dangling references
- return contract violations
- token collisions or semantic ambiguity supported by artifact evidence
- cross-index inconsistencies between callable contracts, token groups, and references

**ALL REPORTED CONTRADICTIONS MUST PASS Q1/Q2/Q3:**

| Test | Meaning |
| ---- | ------- |
| Q1   | Do A and B apply to the SAME operation or entity? |
| Q2   | Do A and B apply in the SAME execution context, scope, or trigger? If context/scope is unspecified or ambiguous:
- do NOT auto-resolve Q2 as YES for Tier 1
- classify the issue as Tier 2 ambiguity unless source verification proves same context |
| Q3   | Is it logically impossible for both A and B to be true simultaneously? |

**REPORTING RULE:**
- Report a contradiction only if Q1 = YES, Q2 = YES, and Q3 = YES.
- If any answer is NO, do NOT report a contradiction.
- Tier 1 requires that the contradiction be reproducible from the same quoted evidence.
- Tier 2 is used when interpretation is required.

**MANDATORY CONTRADICTION EVIDENCE RULE:**
Every reported contradiction MUST include:
  - Quote A (verbatim)
  - Quote B (verbatim)
If both sides cannot be supported by verbatim evidence, the contradiction MUST NOT be reported.

**OUTPUT:**

```
CONTRADICTIONS_TABLE
```

**CLASSIFICATION:**

| Tier   | Meaning                    |
| ------ | -------------------------- |
| Tier 1 | Reproducible contradiction |
| Tier 2 | Potential ambiguity        |


---
### 7. SECTION B — EXECUTION MODEL & RUNTIME FAILURE VALIDATION:

**OBJECTIVE**  
Detect execution failures by validating runtime behavior against the specification’s explicit execution model and persistence rules.

#### STEP 1 — LOAD RUNTIME REASONING CONTEXT  
Load only the segments required for runtime analysis:
- Runtime Execution & UI Flows
- Foundational Rules / Laws
- Persistence / Storage / DB
- all sections where `priority = true`

If one of these categories is not present in `segments.json`, mark it `NOT PRESENT`.

#### STEP 2 — EXTRACT EXECUTION MODEL AXIOMS
From the loaded segments, extract the execution-model axioms explicitly stated in the specification.

**Minimum axiom set to extract when available:**
- what triggers execution or rerun
- what state persists across executions
- when user input values are captured
- what causes state loss or reset

**Rules:**
- If no execution model is explicitly described, create a specification-quality finding for missing execution model.
- In that case only, use `ASSUMED: Standard Sequential Execution` as a temporary analysis baseline.
- If multiple execution models are explicitly described, test each independently.
- Any simulated scenario that contradicts the extracted axioms is invalid and MUST be discarded.

#### STEP 3 — SIMULATE ONLY VALID EXECUTION FLOWS
Simulate runtime behavior only within the loaded context and only under the extracted execution axioms.

The auditor MUST check whether:
- triggers can actually start the expected execution
- required state is still available when later steps depend on it
- user inputs are captured at the correct moment
- reset / rerun behavior destroys data or breaks the intended flow
- persistence logic and runtime flow remain compatible

#### STEP 4 — DETECT EXECUTION FAILURES
Record an execution failure only when the specification implies that the expected runtime behavior cannot succeed, can become inconsistent, or can lose required state.

Each reported execution failure MUST cite:
- at least one execution-axiom quote
- at least one runtime / persistence / specification quote

#### PHASE 2 MUST OUTPUT:
- `EXECUTION_MODEL_AXIOMS`
- `EXECUTION_FAILURE_TABLE`

**EXECUTION REPORTING GUARDRAILS:**
- If an apparent failure depends entirely on framework behavior and not on a specification-controlled decision, do NOT report it
- If the same root cause is already captured in `CONTRADICTIONS_TABLE`, do NOT report it again here
- If the loaded materials contain no explicit ordering, timing, lifecycle, or persistence evidence sufficient for runtime validation, output no execution failure finding and record:
  `INSUFFICIENT EVIDENCE FOR EXECUTION FAILURE VALIDATION`

---
### 8. SECTION C — SPECIFICATION QUALITY CHECK:

**PURPOSE:**  
Detect only **high-signal specification-quality gaps** that materially affect reliable implementation, validation, or safe operation.

**PRIMARY BASELINE:**
- `AUDIT_FRAME`
- `segments.json`
- `contracts.json`
- `EXECUTION_MODEL_AXIOMS`
- verified contradiction / execution evidence already collected in Phase 2

**TOKEN-DISCIPLINE RULES:**
- This section MUST be targeted, not exhaustive.
- Evaluate only gaps that are concrete, source-grounded, and high-impact.
- Do NOT perform a broad best-practices review or style critique.
- Prefer one finding per root cause.
- Reuse already-loaded artifacts and runtime context before reading any additional source text.

**MANDATORY CHECKS:**
1. Goal / scope mismatch visible from `AUDIT_FRAME`
2. Missing implementation-critical inputs, outputs, states, or acceptance criteria
3. Missing or non-measurable testability conditions
4. Missing error behavior when failure handling is required by the stated flow
5. Maintainability or coupling risks only when they create a concrete execution, validation, or change-risk consequence

These checks may produce:
- Tier 1 findings when the gap blocks reliable implementation, execution, or validation
- Tier 2 findings when the gap is advisory only

**TESTABILITY CHECK:**
The auditor MUST verify whether key requirements are testable.
Flag when:
- acceptance criteria are absent for critical flows
- expected outputs are not measurable
- error behavior for critical flows is unspecified
- completion cannot be objectively verified

**REPORTING RULE:**
Do NOT report the same gap separately under multiple categories.

### 2. REPORTING PIPELINE (MANDATORY):

#### Step 1 — Tier Routing
**OBJECTIVE:**
Separate all incoming findings into:
  - Tier 1 findings
  - Tier 2 findings

**TIER 1 FINDINGS** PROCEED THROUGH 
  - the full reporting pipeline.
 
**TIER 2 FINDINGS** PROCEED ONLY THROUGH:
  - Design Intent Filter
  - Severity Assignment
  Then route to advisory output only.

#### Step 2 — Design Intent Filter
**OBJECTIVE:** 
For every finding, verify whether the flagged behavior is:
  - explicitly excluded,
  - explicitly compensated for through a linked safeguard,
  - or explicitly justified in the same section through a "Rationale:", "Design Intent:", or "Justification:" statement.

If yes: 
  - remove the finding from final findings
  - and place it in `ACCEPTED_BY_DESIGN`.

#### Step 3 — Severity Assignment
**OBJECTIVE:** 
Assign severity to all surviving findings.

**DEFAULT SEVERITY MAPPING:**
- Execution Failures, Tier 1 → CRITICAL
- API-Level Conflict, Tier 1 → HIGH
- Return Contract Violation, Tier 1 → HIGH
- Integration Gap, Tier 1 → HIGH
- Broken Reference, Tier 1 → MEDIUM
- Token Collision, Tier 1 → MEDIUM
- All Tier 2 findings → LOW

Override default severity only with explicit justification.

#### Step 4 — Confidence Assignment
**OBJECTIVE:**
Assign a confidence level to every surviving finding.

**ALLOWED VALUES:**
- HIGH = directly supported by explicit artifact/source evidence with minimal interpretation
- MEDIUM = supported by evidence but requires limited interpretation
- LOW = plausible concern but evidence is partial, ambiguous, or assumption-dependent

Confidence MUST be reported in the final finding entry.

#### Step 5 — Source Re-Verification
**OBJECTIVE:** 
Any finding whose evidence is incomplete, source-missing, or explicitly flagged for verification MUST be checked against the original specification text for the referenced section only.

**IF VERIFIED:**
- keep the finding in final findings

**IF NOT VERIFIED:**
- move the finding to `ESCALATED_TO_ARCHITECTURE` with rationale:
  "Source Verification Failed: Referenced text not found in Specification Chunks"

#### Step 6 — Atomic Fix Generation
**OBJECTIVE:**
Generate atomic fixes for Tier 1 findings only.

**ATOMIC FIX REQUIREMENTS:**
- smallest edit surface possible
- exact target location
- operation = ADD / REMOVE / REPLACE
- exact target context
- exact fix content

Tier 2 findings MUST NOT receive atomic fixes.

**TOKEN COLLISION TIE-BREAKER:**
If a Tier 1 finding is a token collision:
- prefer the variant used in callable contracts
- otherwise prefer the variant most consistently used across priority sections

#### Step 7 — Fix Verification

**OBJECTIVE:**
Before finalizing each Tier 1 fix, verify that the proposed fix text does not create a new contradiction against artifact quotes tied to the same entity or section.
Conflict checking MUST use existing artifact quotes tied to the same entity, contract, or section; do not invent hypothetical downstream conflicts.

**IF CONFLICT IS DETECTED:**
  - expand the fix once, OR
  - escalate to `ESCALATED_TO_ARCHITECTURE`

**IF NO CONFLICT IS DETECTED:**
  - mark the fix as `VERIFIED`

**PRIORITY-SECTION REGRESSION FLAG:**
  a) If a VERIFIED or EXPANDED atomic fix modifies any section marked `priority = true` in `segments.json`, the finding MUST also be marked: `REQUIRES_REGRESSION_TEST`
    - This flag does not invalidate the fix.
    - It signals that the modified specification should be re-audited in a later run because the change touches a high-risk section.
  b) In `TIER_1_FINDINGS`, each finding MUST contain:
    - `fix_status`: one of `VERIFIED`, `EXPANDED`, `ESCALATED`
    - `regression_flag`: `REQUIRES_REGRESSION_TEST` (additional flag, not a replacement for fix status) or `NONE`

#### Step 8 — Deduplication and Blocking Status
**OBJECTIVE:**
If the same root cause appears multiple times, keep only the highest-severity instance.

**FINAL BLOCKING STATUS:**
  - BLOCKING = any Tier 1 finding with severity CRITICAL or HIGH
  - NON-BLOCKING = only Tier 1 MEDIUM/LOW findings remain
  - CLEAN = zero Tier 1 findings remain

Tier 2 findings never affect blocking status.

### 3. OUTPUTS

Produce:

1. `FINAL_AUDIT_REPORT` — the only report
2. `audit_fixes.json` — supporting machine-readable fix artifact

**Objective:**
- `FINAL_AUDIT_REPORT` = readable by both human and AI; contains findings, decisions, and implementation status
- `audit_fixes.json` = contains only Tier 1 atomic fixes for downstream AI-coder use

**`FINAL_AUDIT_REPORT` must contain:**

1. `EXECUTIVE_STATUS`
   - `BLOCKING_STATUS`: BLOCKING / NON-BLOCKING / CLEAN
   - `IMPLEMENTATION_READINESS`:
     - READY
     - READY WITH MINOR FIXES
     - NOT READY

2. `TIER_1_FINDINGS`
   - structured findings with verified atomic fixes

3. `ADVISORY_NOTES`
   - Tier 2 findings, never blocking

4. `ACCEPTED_BY_DESIGN`
   - findings dismissed by the Design Intent Filter

5. `ESCALATED_TO_ARCHITECTURE`
   - findings requiring non-atomic change or lacking source verification

6. `RISK_HOTSPOTS`
   - sections ranked by density / priority

Decision must consider:
- remaining Tier 1 findings
- missing validation criteria
- unresolved architectural escalations
- extraction reliability

**MANDATORY FINDING FIELDS**

For every finding:
- id
- location
- category
- tier
- severity
- evidence_quote
- issue
- impact

For Tier 1 findings only:
- operation = ADD / REMOVE / REPLACE
- exact_fix_text
- fix_status
- regression_flag

**FIELD RULES**
- `fact` = observation supported by quoted evidence
- `inference` = diagnosis derived from the fact
- `impact` = execution, validation, or maintenance consequence
- `recommended_action` = concise correction intent
- `exact_fix_text` = atomic patch content for Tier 1 only

**`audit_fixes.json` requirements**
- Tier 1 findings only
- one entry per verified or expanded atomic fix
- each entry must contain:
  - `ref`
  - `tier`
  - `severity`
  - `location`
  - `operation`
  - `target_text` or `target_context`
  - `fix_text`
  - `status`

**Consistency rule:**
- `audit_fixes.json` must match `FINAL_AUDIT_REPORT`
- if there is any mismatch, `FINAL_AUDIT_REPORT` prevails

## 8. CORE RULES (enforced in all phases):

### Rule 1 — Evidence and Missing Information

If required information for execution, validation, scope, or safety is absent, the auditor MUST report it as a finding.  
The auditor MUST NOT silently fill the gap through assumption.

Any assumption used for reasoning must be explicitly labeled `ASSUMED`.

---
### Rule 2 — Fact / Inference / Assumption Separation

Each reported finding MUST distinguish:
- FACT: directly supported by verbatim artifact or source quote
- INFERENCE: logical conclusion derived from facts
- ASSUMPTION: missing information temporarily assumed for reasoning

A finding MUST NOT be reported as Tier 1 if its core conclusion depends on an unstated assumption that was not source-verified.

---
### Rule 3 — Tiering and Contradictions

Each finding MUST be classified as: `Tier 1` or `Tier 2`.

All contradiction findings MUST pass the `Q1/Q2/Q3` test.

---
### Rule 4 — Reproducibility

Tier 1 findings MUST be reproducible from the same quoted evidence.
If reproducibility depends on interpretation, classify the finding as Tier 2.

---
### Rule 5 — Phase Boundary and Efficiency

This process is a single-run 3-phase audit pipeline.  
Each phase may access only its declared inputs.  
The auditor MUST NOT speculatively reload the full specification.

**If required inputs are missing:**
- output `INSUFFICIENT INPUTS`
- stop the audit at that phase

**If a prior artifact is internally inconsistent:**
- explicitly record the inconsistency
- repair or re-derive it before continuing

---
### Rule 6 — Evidence Hard Gate

The auditor MUST use only the inputs explicitly declared for the current phase.

Rules:
- No quote = no finding
- Any ambiguous but relevant evidence MUST be marked `UNCERTAIN`
- If a later-phase finding depends on earlier artifacts, verify against the extraction artifacts first
- Do not rely on prior chat context unless it is explicitly part of the current phase inputs


---
### Rule 7 — Scope-Proportional Judgment

The auditor MUST prioritize correctness, execution reliability, and testability.  
It MUST NOT recommend architectural sophistication unless required to fix a concrete risk, contradiction, or execution failure.

The auditor MUST evaluate findings against the intended artifact scope and maturity level visible in the source or `AUDIT_FRAME`.

The auditor MUST distinguish:
- true blocking gaps
- important implementation ambiguities
- optional improvements

The auditor MUST NOT report a finding solely because the design is less sophisticated than a possible alternative.

### Rule 8 — Finding Finality by Phase

Findings produced during analysis MUST NOT be silently suppressed mid-phase.

They may only be:
- filtered by the Design Intent Filter in Phase 3
- merged during explicit deduplication
- escalated when atomic-fix verification fails
