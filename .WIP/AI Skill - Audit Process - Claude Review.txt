## Core Insight: The Token Problem Is Architectural

In Claude.ai, every pass accumulates in context. With 6 passes on a 120k-char spec:

| Turn | Approx. input context |
|---|---|
| Pass 0 | 120k (spec) + 3k (instructions) |
| Pass 1 | 120k + 3k + 2k (Pass 0 output) + 5k (instructions) |
| Pass 2 | 120k + all above + 8k (Pass 1 output) + 3k |
| ... | Keeps growing |
| Pass 5 | 120k + ~35k (all outputs) + ~25k (all instructions) = **~180k** |

**Total tokens consumed across 6 turns: ~750k–900k.** Most of that is re-reading the spec and prior outputs.

The highest-leverage optimization isn't scripts that shave 5k off the spec — it's **reducing the number of turns and keeping the full spec in context for fewer of them.**

---

## Proposed Architecture: 3-Phase, 4-Turn Pipeline

```
SPEC
  ↓
Phase 1: EXTRACT  (1–2 LLM turns, spec in context)
  ↓
Script: COMPILE   (mechanical, no LLM)
  ↓
Phase 2: ANALYZE  (1 LLM turn, artifacts only — spec NOT re-read)
  ↓
Phase 3: REPORT   (1 LLM turn, findings + artifacts)
  ↓
FINAL AUDIT REPORT
```

### Why This Works

| Design lever | Effect |
|---|---|
| 3–4 turns instead of 6 | ~40% fewer context-accumulation cycles |
| Spec loaded in Phase 1 only | Phases 2–3 work from ~5k of JSON artifacts, not ~120k of spec |
| Script between phases | Mechanical work done at zero LLM cost; pre-flags candidates |
| JSON artifacts on disk | Machine-parseable, compact, decoupled from conversation |

**Estimated total tokens: ~280k–350k** (vs. ~750k–900k current). A **55–65% reduction.**

## Phase Design

### Phase 1 — EXTRACT (replaces Pass 0 + 1 + 2)

**Turns:** 1 for specs under ~60k chars. 2 for larger specs (split: Turn A = structure + tokens, Turn B = contracts).

**Why merge?** All three extraction tasks need the full spec. Running them in fewer turns means the spec occupies context fewer times. The extraction tasks are independent enough for the LLM to produce them in sequence within one response.

**Critical change: output to JSON files, not markdown tables in conversation.**

Currently, Pass 1 produces 3 large markdown tables (~8k tokens) that sit in the conversation forever. Instead:

1. LLM extracts artifacts
2. LLM writes each artifact to `/home/claude/audit_artifacts/<name>.json`
3. Conversation contains only a **brief summary** (counts, key stats)

This means Phases 2–3 read from compact files, not from 15–20k tokens of tables in conversation history.

**Outputs (JSON files):**

| Artifact | Content |
|---|---|
| `segments.json` | Section map with IDs, line ranges, priority flags |
| `contracts.json` | Function signatures, return types, consumption sites, cross-references |
| `tokens.json` | Concept groups with variant lists and locations |

**Conversation output:** ~500 tokens summary ("Extracted 24 segments, 8 priority. 30 callable signatures. 15 concept groups, 4 with variant ambiguity.")

### Script — `compile_artifacts.py`

**This is where I'd use a script, and here's why it's different from the mini-spec's approach.**

The mini-spec proposed scripts to parse the **input spec** (variable format, fragile). I propose a script that parses **LLM-generated JSON artifacts** (controlled format, reliable).

**What it does:**

1. **Parse** the 3 JSON artifact files (reliable — we control the schema)
2. **Mechanical conflict detection** (zero LLM cost):
   - Duplicate function names with different signatures
   - Dangling cross-references (target not found in segments)
   - Token groups with 3+ variants (high collision risk)
   - Functions declared but never consumed (or consumed but never declared)
   - Return type declared ≠ return type expected at consumption site (string comparison)
3. **Risk scoring** per section: count of incoming/outgoing references × priority flag
4. **Output:** `compiled_audit.json` containing:
   - `conflict_candidates[]` — pre-flagged items with mechanical evidence
   - `risk_hotspots[]` — sections ranked by cross-reference density
   - `artifact_stats{}` — counts for audit coverage validation

**Why this script is viable:**
- Input is JSON (we control the schema) — no format variability
- Operations are pure string comparison, counting, set intersection — zero ambiguity
- Output is compact (~1–2k tokens vs. ~15–20k of raw artifacts)
- Focuses the LLM's analysis budget on pre-flagged candidates

**Token impact:** The LLM in Phase 2 receives `compiled_audit.json` (~2k tokens) as its primary input instead of scanning 3 raw artifact tables (~15k tokens). It can still read the full JSON files when it needs verbatim quotes, but the script tells it **where to look**.

### Phase 2 — ANALYZE (replaces Pass 3 + 4)

**Turns:** 1.

**Why merge Pass 3 + 4?** The script has already done the scanning work. The LLM's job is now **judgment on pre-flagged candidates**, not exhaustive scanning. This is a smaller cognitive load that fits in one turn.

**Inputs:**
- `compiled_audit.json` (loaded into context — ~2k tokens)
- `segments.json`, `contracts.json`, `tokens.json` (read from disk as needed for verbatim quotes)
- **NOT the full spec** — this is the key savings

**Two sections executed sequentially in one turn:**

**Section A — Contradiction Analysis:**
- For each candidate in `conflict_candidates`: apply Q1/Q2/Q3 test
- Fetch verbatim quotes from JSON artifacts (not from spec)
- Classify Tier 1 / Tier 2
- Produce `CONTRADICTIONS_TABLE`

**Section B — Execution Model Validation:**
- Extract execution axioms from spec chunks tagged as "Runtime" or "Laws" in `segments.json`
- `[Assumed]` For this section only, load relevant spec chunks (not full spec — only sections flagged as runtime/lifecycle in segments.json)
- Simulate execution flows against axioms
- Produce `EXECUTION_FAILURE_TABLE`

**Spec chunk loading for Phase 2:** This is a nuanced point. Phase 2 needs some original spec text for execution model validation (axiom extraction, sequence verification). But it doesn't need the full spec. The instruction would be: "Read ONLY sections classified as 'Runtime', 'Laws', or 'Persistence' in `segments.json`." The LLM fetches specific sections by line range from the uploaded file.

### Phase 3 — REPORT (replaces Pass 5)

**Turns:** 1. Same logic as current Pass 5 but with one addition.

**Inputs:**
- Findings from Phase 2 (in conversation)
- `compiled_audit.json` for cross-validation
- Artifact JSONs for verbatim quotes in fixes

**Outputs:**
- `FINAL_AUDIT_REPORT` (in conversation + written to file)
- `audit_fixes.json` (machine-readable fixes for downstream AI-coder consumption)

The `audit_fixes.json` is new — it structures atomic fixes as a JSON array that could be programmatically applied:

```json
[
  {
    "ref": "C-001",
    "tier": 1,
    "severity": "HIGH",
    "location": "§4.7 §3.2",
    "operation": "REPLACE",
    "target_text": "Return UUID or None",
    "fix_text": "Return project_id (UUID) or None",
    "status": "VERIFIED"
  }
]
```

This aligns with your workflow: audit → fix → feed to AI-coder.

## Script Specification: `compile_artifacts.py`

```
Input:  /home/claude/audit_artifacts/{segments,contracts,tokens}.json
Output: /home/claude/audit_artifacts/compiled_audit.json (stdout summary)

Operations (all mechanical, zero ambiguity):
1. Parse JSON files (try/except per file — skip on failure)
2. DUPLICATE_FUNCTIONS: group by function name, flag where signatures differ
3. DANGLING_REFS: cross-reference targets against segments.section_ids
4. RETURN_MISMATCHES: compare declared return type vs consumer expected type (string equality)
5. TOKEN_COLLISIONS: flag concept groups with 3+ variants
6. ORPHAN_FUNCTIONS: functions with 0 consumption sites
7. RISK_DENSITY: per-section count of (incoming refs + outgoing refs) × priority_weight
8. Write compiled_audit.json with all candidates + stats
```

**Fault tolerance:** Each operation is independent. If one fails (malformed JSON field), it logs a warning and continues. The compiled output includes a `parsing_warnings[]` array.
