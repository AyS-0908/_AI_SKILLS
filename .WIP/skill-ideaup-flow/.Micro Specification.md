# IDEASUP FLOW: PROMPT BASED APPROACH PRESENTATION

## PRESENTATION:

<Project>
IDEASUP FLOW is a prompt suite for studying and preparing startup creation.
Each prompt is a scoped pipeline stage: Pain -> Opportunity -> Idea -> Business Plan -> User Story -> Mockup -> Specification -> AI-Coder rules
</Project>

<Source_Of_Truth>
- Treat each stage file as bounded by its guideline scope contract.
- Do not import assumptions from adjacent stages unless the guideline explicitly allows it.
</Source_Of_Truth>

<Stage_Contracts>

| Stage | Input(s) | Owns | Excludes |
|---|---|---|---|
| Pain / Need | [to clarify] | Customer pains / needs; scoring | Ideas; business plan; specification |
| Opportunity | Pain / Need | Macro-ideas; macro-assessment | Idea details; business plan; specification |
| Idea | Opportunity | Idea definition; target user; problem; solution concept; key assumptions; differentiation; initial scope direction | Business plan; user story; UI; mockup; specification; technical choices |
| Business Plan | Idea | Market logic; positioning; business model; revenue logic; GTM assumptions; viability | User story; mockup; specification; technical implementation |
| User Story | Idea | Persona; problem; value hypothesis; V1 scope rationale; V2/V3 candidates | Screens; components; data model; business rules; edge cases; tech |
| Mockup | User Story | Screens; layout; components; navigation; interaction patterns; visible data shapes; UI states; demo data; Mockup Brief | V2/V3; business rules; edge cases; persistence; stack; implementation logic |
| Specification | User Story + optional Mockup | Business rules; edge cases; persistence model; feature-specific external services; acceptance criteria; non-goals; privacy/risk notes; tool stack; auth; key technical constraints | Persona/problem/idea duplication; UI layout/components duplication; language/framework/DB choice |
| AI-Coder rules | Standalone project setup | Language; framework; DB; repo layout; naming; commit style; test policy; project-wide always/never rules | Feature-specific requirements; business rules; UI decisions; product scope |
</Stage_Contracts>

<Hard_rules>
- Later stages reference upstream artifacts; they do not duplicate them.
- Excluded content must not be generated by inference.
- AI-Coder consumes only: `Specification`, `Mockup HTML` if available, and `AI-Coder rules`.
</Hard_rules>

<Work_Rules>
- Preserve AI-native structure: XML-like tags first (Markdown only when useful).
- Prefer compact instructions over explanatory prose.
- Keep stage ownership MECE: no overlap with previous or next prompts.
- Check Ripple Effect for upfront/downstream stages when relevant.
- Separate facts, assumptions, questions, outputs, and navigation.
- Do not add speculative abstractions, hidden flexibility, or generic startup advice.
</Work_Rules>
