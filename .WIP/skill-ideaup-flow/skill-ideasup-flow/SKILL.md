# IdeaUp Flow

## Gotchas

- Skill type: RIGID. Execute only the requested stage and its declared layer.
- Stage 4 Business Plan is planned but `MISSING SOURCE`; Stage 8 AI-Coder rules is planned but `MISSING SOURCE`.
- Do not execute missing stages unless the user provides their missing source.
- Do not invent stage logic from general startup knowledge, adjacent stages, or memory.
- Pain, Opportunity, and Idea evidence checks may require browsing. If browsing or source access is unavailable, say so before making evidence-backed claims.
- Remove ChatGPT Project, GPT Action, Google Sheets, copy-chat navigation, install, deployment-wrapper, and prompt-registry packaging from stage outputs unless the user explicitly asks for packaging work.
- Re-check near-miss prompts before using this skill: "create a GPT Action", "write investor deck", "build the app", and "make a generic startup idea list" are not IdeaUp stage execution requests by default.

## Pipeline Contract

Stage order:

1. Pain
2. Opportunity
3. Idea
4. Business Plan - `MISSING SOURCE`
5. User Story
6. Mockup
7. Specification
8. AI-Coder rules - `MISSING SOURCE`

Stage ownership:

| Stage | Owns | Excludes |
|---|---|---|
| Pain | Customer pains and needs, evidence, clustering, scoring | Ideas, business plan, specification |
| Opportunity | Macro-opportunities, macro-assessment, shortlist | Idea detail, business plan, specification |
| Idea | Idea definition, target user, problem, solution concept, key assumptions, differentiation, initial scope direction | Business plan, user stories, UI, mockup, specification, technical choices |
| Business Plan | `MISSING SOURCE` | Do not infer |
| User Story | Persona summary from Idea, problem summary from Idea, value hypothesis, V1 scope rationale, V2/V3 candidates | Screens, components, data model, business rules, edge cases, technical implementation |
| Mockup | Screens, layout, components, navigation, interaction patterns, visible data shapes, UI states, demo data, Mockup Brief | V2/V3, business rules, edge cases, persistence, stack, implementation logic |
| Specification | Business rules, edge cases, persistence model, feature-specific external services, acceptance criteria, non-goals, privacy/risk notes, tool stack, auth, key technical constraints | Persona/problem/idea duplication, UI layout/components duplication, language/framework/database choice |
| AI-Coder rules | `MISSING SOURCE` | Do not infer |

Anti-overlap rules:

- Later stages reference upstream artifacts; they do not duplicate them.
- Excluded content must not be generated by inference.
- If a requested output crosses stages, name the boundary and ask the user which stage to run.
- Keep facts, assumptions, questions, and outputs separate.
- Prefer narrow, testable V1 decisions over broad product vision.

Source priority:

1. Current user instruction.
2. Validated upstream stage artifact.
3. Validated answers collected in the current stage.
4. Public evidence when the stage requires evidence.
5. Labeled inferred defaults.

If sources conflict, ask a minimal clarification. Do not silently resolve contradictions.

Ripple Effect rule: when stage order, file names, ownership, or missing-stage status changes, update this contract, the reference map, and every affected stage reference together; then re-run the near-miss trigger tests.

## Execution

1. Identify the requested stage. If ambiguous, ask the user to choose one available stage.
2. If the stage is missing, state `MISSING SOURCE` and ask the user to provide the source before continuing.
3. Read only the matching reference file.
4. Run the reference workflow exactly. Ask only the minimum stage questions, using inferred defaults when safe.
5. Stop for user validation wherever the reference says to stop.
6. Use compact, AI-native Markdown/XML-style structure when useful.
7. Do not add copy-chat navigation, install instructions, or unrelated packaging notes to stage outputs.
