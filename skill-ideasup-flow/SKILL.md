# IdeaUp Flow

## Gotchas

- Skill type: RIGID. Execute only the requested stage and its declared layer.
- Stage 4 Business Plan and Stage 8 AI-Coder rules are planned but `MISSING SOURCE`. Do not execute or invent them unless the user provides their source.
- Do not invent stage logic from general startup knowledge, adjacent stages, or memory.
- Pain, Opportunity, and Idea evidence checks may require browsing. If browsing is unavailable, say so before making evidence-backed claims.
- Artifact files may be absent, empty, or hand-edited. An empty or malformed upstream artifact = missing input: output the stage's `MISSING DATA` message and stop.
- Keep GPT Action, Google Sheets, copy-chat navigation, install, and deployment packaging out of stage outputs.

## Pipeline Contract

Stage order and artifacts (all under `<project>/ideasup/`):

| # | Stage | Artifact | Input artifact |
|---|---|---|---|
| 1 | Pain | `1-pain.md` | user scope |
| 2 | Opportunity | `2-opportunity.md` | `1-pain.md` |
| 3 | Idea | `3-idea.md` | `2-opportunity.md` |
| 4 | Business Plan | `MISSING SOURCE` | — |
| 5 | User Story | `5-user-story.md` | `3-idea.md` |
| 6 | Mockup | `6-mockup.html` | `5-user-story.md` |
| 7 | Specification | `7-specification.md` | `5-user-story.md` + optional `6-mockup.html` |
| 8 | AI-Coder rules | `MISSING SOURCE` | — |

Stage ownership:

| Stage | Owns | Excludes |
|---|---|---|
| Pain | Customer pains and needs, evidence, clustering, scoring | Ideas, business plan, specification |
| Opportunity | Macro-opportunities, macro-assessment, shortlist | Idea detail, business plan, specification |
| Idea | Idea definition, target user, problem, solution concept, wedge, key assumptions, differentiation, initial GTM channels, initial scope direction | Business plan, user stories, UI, mockup, specification, technical choices |
| User Story | Persona summary from Idea, problem summary from Idea, value hypothesis, V1 scope rationale, V2/V3 candidates | Screens, components, data model, business rules, edge cases, technical implementation |
| Mockup | Screens, layout, components, navigation, interaction patterns, visible data shapes, UI states, demo data, Mockup Brief | V2/V3, business rules, edge cases, persistence, stack, implementation logic |
| Specification | Business rules, edge cases, persistence model, feature-specific external services, acceptance criteria, non-goals, privacy/risk notes, tool stack, auth, key technical constraints | Persona/problem/idea duplication, UI layout/components duplication, language/framework/database choice |

Anti-overlap rules:

- Later stages reference upstream artifacts; they do not duplicate them.
- Excluded content must not be generated by inference.
- If a requested output crosses stages, name the boundary and ask the user which stage to run.
- Keep facts, assumptions, questions, and outputs separate.
- Prefer narrow, testable V1 decisions over broad product vision.

Source priority:

1. Current user instruction.
2. Validated upstream stage artifact file.
3. Validated answers collected in the current stage.
4. Public evidence when the stage requires evidence.
5. Labeled inferred defaults.

If sources conflict, ask a minimal clarification. Do not silently resolve contradictions.

Ripple Effect rule: when stage order, file names, ownership, or missing-stage status changes, update this contract, the reference map, `scripts/status.py`, and every affected stage reference together.

## Deterministic Checks

Scripts verify mechanics; content quality stays with the stage references.

- `scripts/status.py [artifact_dir]` — pipeline status and next stage.
- `scripts/check_mockup.py <6-mockup.html>` — run after generating or patching the mockup HTML, before presenting it: single style/script tag, no external dependencies, Mockup Brief comment with all 5 fields, balanced DEMO ONLY markers, no persistence/network APIs. Fix failures and re-run until PASS.
- `scripts/check_spec.py <7-specification.md>` — run on the drafted final Specification before presenting it: all required sections present, Do Not Build item count in range.

## Artifacts And Resume

- Artifact dir: `ideasup/` in the user's current project (create on first stage completion). If the user names another dir, use it.
- "status", "where are we", "continue", "resume", or "run the flow" → run `python scripts/status.py <artifact_dir>` and report; for continue/resume/flow, then run the NEXT stage it names.
- Stage input: read the input artifact file from the table above. Only ask the user to paste input if the file is absent.
- Stage output: after the user validates the stage's last deliverable, write that validated deliverable verbatim to the stage's artifact file, then state the next stage (skipping `MISSING SOURCE` stages).
- Flow mode ("run the flow"): run stages in order through the same per-stage workflow. Every validation stop in the references still applies — flow mode chains stages, it never skips validation.
- Re-running a completed stage overwrites its artifact only after user confirmation; warn that downstream artifacts may become stale (Ripple Effect).

## Execution

1. Identify the requested stage (or flow/status request). If ambiguous, ask the user to choose one available stage.
2. If the stage is `MISSING SOURCE`, say so and ask for the source before continuing.
3. Read only the matching reference file, and the stage's input artifact if it exists.
4. Run the reference workflow exactly. Ask only the minimum stage questions, using inferred defaults when safe. For the output-language question, offer the user's message language as the inferred default.
5. Stop for user validation wherever the reference says to stop.
6. On final validation, persist the artifact per Artifacts And Resume.
7. Use compact, AI-native Markdown/XML-style structure when useful. No copy-chat navigation or packaging notes.
